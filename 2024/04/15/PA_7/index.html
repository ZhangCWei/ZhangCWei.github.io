Interproducedural Analysis

Motivation

  • So far, all analyses we learnt are intraprocedural (过程内的). How to deal with method calls?
    • Make the most conservative (保守的) assumption for method calls, for safe-approximation (过程内分析对函数调用做最保守的估计)
    • Source of imprecision
      • x=NAC, y=NACx=NAC,\ y=NACx=NAC, y=NAC
      • n=NACn=NACn=NAC
  • For better precision, we need Interprocedural analysis: propagate data-flow information along interprocedural control-flow edges
    • 过程间分析：沿着过程间控制流边传播数据流信息。如 call and return edges
    • x=42, y=43x=42,\ y=43x=42, y=43
    • n=10n=10n=10
  • To perform interprocedural analysis, we need call graph.

Call Graph Construction (CHA)

  • A representation of calling relationships in the program
    • Essentially, a call graph is a set of call edges from call-sites to their target methods (callees)

Applications of Call Graph

  • Foundation of all interprocedural analyses
  • Program optimization
  • Program understanding
  • Program debugging
  • Program testing
  • And many more …

Call Graph Construction for OOPLs (focus on Java)

  • Class hierarchy analysis (CHA)
  • Rapid type analysis (RTA)
  • Variable type analysis (VTA)
  • Pointer analysis (k-CFA)

Method Calls (Invocations) in Java

  • Java中的调用主要分为三大类，对应了四种指令，Java8开始还引入了invokedynamic，主要用于实现动态类型语言的，在这里我们不进行讨论。
    • invokestatic: 调用的目标方法是 static methods (静态方法，它们属于类本身，不属于类的特定实例)，故没有 reciever object，目标个数为一个，可在编译期确定。
    • invokespecial: 调用构造函数、调用私有的实例方法、调用父类的实例方法，它的目标个数也是只有1个，在编译期可以确定。
    • invokeinterface, invokevirtual: 调用其他的实例方法，因为有多态的存在，所以可能调用不同的方法，目标方法可能大于一个，具体调用的方法要在运行时确定。
  • Virtual call is the key to call graph construction for OOPLs

Method Dispatch of Virtual Calls

  • During run-time, a virtual call is resolved based on
    • type of the receiver object (pointed by o) : 𝑐
    • method signature at the call site : 𝑚
  • In this lecture, a signature acts as an identifier of a method (签名唯一确定方法)
    • 定义方法的类、方法名、Descriptor (包括返回类型和参数类型)
  • We define function Dispatch(𝑐,𝑚)Dispatch(𝑐, 𝑚)Dispatch(c,m) to simulate the procedure of run-time method dispatch (这个函数模拟动态时 Method dispatch 的过程)
    • 如果 𝑐 类中包含一个非抽象的方法 𝑚′，并且其与 𝑚 拥有一样的 name 和 descriptor，那么 𝑚′ 就确定为这个方法，被函数返回。
      • 注意，调用的方法一定有函数体，所以是非抽象的
    • 如果 𝑐 类中没有符合条件的方法，那么就会对 𝑐 的父类重复这个 Dispatch 的过程。
  • Example
    • Dispatch(B, A.foo()) = ?
      • A.foo()
    • Dispatch(C, A.foo()) = ?
      • C.foo()

Class Hierarchy Analysisi* (CHA)

  • Require the class hierarchy information (inheritance structure) of the whole program. CHA 方法需要类层次结构信息 (继承信息)
  • Resolve a virtual call based on the declared type (声明类型) of receiver variable (接收变量) of the call site. 下图的 call site 为 a，其声明类型为 A
  • Assume the receiver variable a may point to objects of class A or all subclasses of A. 假设变量 a 可以指向 A 类以及所有 A 的子类的对象，CHA 去解目标方法的过程就是去查询 A 类的继承结构 class hierarchy。
    • Resolve target methods by looking up the class hierarchy of class A

Call Resolution of CHA

  • We define function Resolve(cs)Resolve(cs)Resolve(cs) to resolve possible target methods of a call site cscscs by CHA, mmm = method signature at cscscs
    • 传入的参数为调用点 cscscs
    • 初始化 TTT 为空，存放目标方法
    • 用 mmm 保存 cscscs 点处的方法签名
  • For static call, 目标方法就是写在call site中的方法，签名就是 mmm。如下图，直接用类 C 调用方法而不需要定义对象 c，很明显调用的就是 C 的方法，所以可直接加到集合T中。
  • For special call, 需要处理构造函数、私有方法、父类方法三种情况。
    • 以父类方法为例，如图所示，C 类继承 B 类，C 类里有一个 super 调用，这个 foo() 虽然在当前类有定义，但是实际使用的是父类 B 的 foo()，因此需要使用Dispatch函数 (因为 B 类不一定就有这个 foo 方法，B 继承自 A，foo 方法不一定是 B 的，所以需要再用dispatch)，其中的 foo() 的签名 m 由编译器返回信息可以知道是 B 的，那么获取 foo() 返回值的 c 也指向 B，也就相当于在父类中寻找了。
    • 对于私有方法和构造函数，目标方法其实与 static call 一样，就是写在签名里的 m
  • For virtual call, 首先会取出 cscscs 位置上的接收变量的声明类型 (declared type)，这里是C，接下来会对 C 及其继承树上的所有子类，这里定义为c’，调用 Dispatch(c′,m)Dispatch(c', m)Dispatch(c′,m)，并将所有的结果都加到T中。
  • Example

    • 如下图所示，有4个类，其中 A 有foo方法，C、D 都复写了foo方法

    • 有3个call site，如下图所示，求每个call site的目标方法

      • Resolve(c.foo()) = {C.foo()} ，这里是virtual call，所以会把 C 及其子类进行dispatch，又因为 C 没有子节点，所以只需要对 C 进行dispatch（dispatch 的作用是遍历这个 class 看看有没有对应函数，没有的话再遍历其父类），C有foo，不需要遍历其父节点，所以答案为{C.foo()}
      • Resolve(a.foo()) = {A.foo(), C.foo(), D.foo()} ，同理，会把 A 及其子类进行 dispatch，也就是对 A B C D 进行dispatch，对A dispatch的结果就是A，对B dispatch的时候，B 没有 foo，所以会找其父类，结果也是 A，再对 B 的子类 C D 分别进行dispatch，结果加入 C D，所以答案为 A C D
      • Resolve(b.foo()) = {A.foo(), C.foo(), D.foo()} ，这里会先对 B, C, D 分别进行 dispatch，对 B dispatch 的时候，B 没有 foo，所以会找到其父类 A，A 有 foo()，所以结果加入 A，再对 C D 进行 dispatch，同理，答案也是 A C D

    • 假如第3个情况中，指定 B b = new B() 呢？

      • CHA 只考虑变量的声明类型，哪怕写了B b = new B()，这里还是会根据其声明类型 B 来解目标方法，所以答案仍是A C D
      • 这也是 CHA 的一个问题，C.foo() 和 D.foo() 对于这个情况来说是虚假的目标方法 (spurious call targets)。CHA 只考虑声明类型，这样就会导致精度的下降。

Features of CHA

  • Advantage: fast
    • Only consider the declared type of receiver variable at the call-site, and its inheritance hierarchy (只考虑在调用点上接收变量的声明类型及其继承层次结构)
    • Ignore data- and control-flow information (忽略数据和控制流信息)
  • Disadvantage: imprecise
    • Easily introduce spurious target methods (容易导致假的目标方法)
    • Addressed in next lectures
  • Common usage: IDE

Call Graph Construction (via CHA)

  • Build call graph for whole program via CHA
    • Start from entry methods (focus on main method)
    • For each reachable method 𝑚𝑚m, resolve target methods for each call site cscscs in 𝑚𝑚m via CHA (Resolve(cs)Resolve(cs)Resolve(cs))
    • Repeat until no new method is discovered
  • Algorithm
    • 从 main 开始，从 WLWLWL 中取出一个方法 mmm，如果它没有被处理过 (不在 RMRMRM 中)，就对 mmm 中所有的 call site 进行 resolve 分析，获得目标方法集合 TTT，将其中的目标方法放入 WLWLWL 中，并在 CGCGCG 中加上调用边 cs→m′cs → m'cs→m′。然后循环再对 WLWLWL 中的方法进行分析，直到 WLWLWL 为空
    • 类似于带标记的 BFS
  • Example
    • Initialization： WLWLWL = [A.main()]
      • 取出方法 A.main()，将其加入可达集合 RMRMRM 中，求解其中的 call site，即 A.foo()，求得 Resolve(A.foo())={A.foo()}，构造 A.main() → A.foo() 的一条边，并将 A.foo() 加入 WLWLWL 中
    • WLWLWL = [A.foo()]
      • 取出方法 A.foo()，将其加入可达集合 RMRMRM 中，求解其中的 call site (这里不考虑构造函数)，即 a.bar()，求得 Resolve(a.bar())={A.bar(), B.bar(), C.bar()}，构造下图右侧的三条边，并将它们加入 WLWLWL 中
    • WLWLWL = [A.bar(), B.bar(), C.bar()]
      • 同理，取出 A.bar()，将其加入可达集合 RMRMRM 中，Resolve(c.bar()) = {C.bar()}，构造下图左侧内部边，并将 C.bar() 加入 WLWLWL 中
    • WLWLWL = [B.bar(), C.bar(), C.bar()]
      • 同理，取出 B.bar()，其方法体为空，不需要进行操作
    • WLWLWL = [C.bar(), C.bar()]
      • 同理，取出 C.bar()，将其加入可达集合 RMRMRM 中，Resolve(c.bar()) = {A.foo()}，构造下图左侧外部边，并将 A.foo() 加入 WLWLWL 中
    • WLWLWL = [C.bar(), A.foo()]
      • 取出 C.bar()，它已经在 RMRMRM 中了，不需要进行操作
    • WLWLWL = [A.foo()]
      • 取出 A.foo()，它已经在 RMRMRM 中了，不需要进行操作
    • WLWLWL = []
      • 算法结束

Interprocedural Control-Flow Graph

  • CFG represents structure of an individual method 单个方法的结构
  • ICFG represents structure of the whole program 整个程序的结构
    • With ICFG, we can perform interprocedural analysis
  • An ICFG of a program consists of CFGs of the methods in the program, plus two kinds of additional edges:
    • Call edges: from call sites to the entry nodes of their callees
      • 从调用点 call sites 连接到其目标方法 callees 的入口 entry nodes
    • Return edges: from exit nodes of the callees to the statements following their call sites (i.e., return sites)
      • 从 return 语句连接到紧跟着的调用点call sites的下一个语句 (返回点)
  • ICFG = CFGs + call egdes & return edges
    • 连接这两种边的信息来自 call graph
  • 如上图所示，ICFG 就是 CFG 加上两种边，但为何还要保留蓝红虚线之间的原有边呢？
    • 这个边 (从call site 到return site的边) 叫做 call-to-return edge，这个边可以用来在 ICFG 中传播本地数据流 (就是函数内部的数据流)。
    • 如果没有这条边，我们需要跨越其他方法来传播本地数据流，这是非常低效的。

Interprocedural Data-Flow Analysis

  • Analyzing the whole program with method calls based on interprocedural control-flow graph (ICFG)
  • 过程内主要是关注每个CFG，而过程间需要加上调用和返回边，在转换函数中，过程间还需要加上两种边的转换函数edge transfer
Intraprocedural Interprocedural
Program representation CFG ICFG = CFGs + call&return edges
Transfer functions Node transfer Node transfer + edge transfer
  • Call edge transfer: 将数据流从 call node 传送到被调用节点 callee (沿着 call edges，传递参数值)
  • Return edge transfer: 将数据流从 callee 上的 return node 传送到 return site (沿着return edges，传递返回值)
  • Node transfer: same as intraprocedural constant propagation, except that
    • For call nodes, the transfer function is identity function
    • 对于每个调用节点 call node (例如b=addOne(a))，会将等式左侧的数值kill掉，然后在下一步中由返回边传递函数重新赋值。这个操作可以在返回值与原值不同时防止数据冲突。
  • Example
    • 黄色节点表示 IN 数据流，蓝色表示 OUT 数据流
  • Call-to-return edges allow the analysis to propagate local data-flow (a=6 in this case) on ICFG. Without such edges, we have to propagate local data-flow across other methods, which is very inefficient. 如果没有这条边，就需要将令 a 走过一遍调用的函数，会导致低效，甚至出现错误。如下图所示：
  • 将等式左侧的数值kill掉的原因是防止不精确。例如，下图中，如果不进行 kill，那么 b = 7 或 10，最终就成为 NAC，这样就不够精确。

Interprocedural Constant Propagation, In Summary

  • Node transfer
    • Call nodes: identity
    • Other nodes: same as intraprocedural constant propagation
  • Edge transfer
    • Normal edges: identity
    • Call-to-return edges: kill the value of LHS variable of the call site, propagate values of other local variables
    • Call edges: pass argument values
    • Return edges: pass return values
